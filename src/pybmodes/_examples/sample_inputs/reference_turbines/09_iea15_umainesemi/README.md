# IEA-15-240-RWT on the UMaine VolturnUS-S floating semi-submersible

UMaine VolturnUS-S semi-submersible: tower modelled as a free-free flexible beam (hub_conn = 2) with a PlatformSupport block assembled programmatically from the upstream IEA-15 v1.1 OpenFAST decks. The redesigned (v1.1) tower targets a 1st-FA frequency around 0.49 Hz to avoid 3P resonance.

## Files

| File                                       | Purpose                              |
| ------------------------------------------ | ------------------------------------ |
| `09_iea15_umainesemi_tower.bmi`                   | Tower BMI (floating with PlatformSupport)         |
| `09_iea15_umainesemi_tower_sec_props.dat`         | Distributed tower section data       |
| `09_iea15_umainesemi_blade.bmi`                   | Blade BMI (rotating cantilever)      |
| `09_iea15_umainesemi_blade_sec_props.dat`         | Distributed blade section data       |

## How to run

```python
from pybmodes.models import Tower, RotatingBlade

tower = Tower("src/pybmodes/_examples/sample_inputs/reference_turbines/09_iea15_umainesemi/09_iea15_umainesemi_tower.bmi")
tower_modal = tower.run(n_modes=8)
print("tower freqs (Hz):", tower_modal.frequencies[:4])

blade = RotatingBlade("src/pybmodes/_examples/sample_inputs/reference_turbines/09_iea15_umainesemi/09_iea15_umainesemi_blade.bmi")
blade_modal = blade.run(n_modes=8)
print("blade freqs (Hz):", blade_modal.frequencies[:4])
```

## Comparison with published values

The original publication for this RWT printed a 1st-FA tower-bending frequency of **~ 0.4900 Hz** (IEA-15-240-RWT v1.1 release notes (IEAWindTask37/IEA-15-240-RWT): redesigned floating tower target 1st-FA ≈ 0.49 Hz). Reference-wind-turbine structural definitions are **iteratively revised** across releases — the same RWT designation at git-tag v1.0.0 may have a few-percent different section-property distribution than at v2.0.0. The pyBmodes frequency above is derived from the deck-as-distributed at the time this sample was last built, so it need not match the publication's printed value exactly. A drift between them usually reflects deck-revision evolution, not a pyBmodes error — treat the published value as a historical anchor, not a regression target.
